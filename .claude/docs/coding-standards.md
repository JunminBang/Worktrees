# 코딩 표준

- 모든 게임 코드의 공개 API에는 문서 주석 필수
- 모든 시스템에는 `docs/architecture/`에 아키텍처 결정 기록(ADR) 필요
- 게임플레이 수치는 데이터 드리븐(외부 설정 파일) — 하드코딩 금지
- 모든 공개 메서드는 단위 테스트 가능하게 설계 (싱글턴 대신 의존성 주입)
- 커밋 시 관련 설계 문서 또는 태스크 ID 참조 필수
- **검증 주도 개발**: 게임플레이 시스템 추가 시 테스트 먼저 작성.
  UI 변경은 스크린샷으로 검증. 기대 출력과 실제 출력을 비교한 후 완료 처리.
  모든 구현에는 작동 증명 수단이 있어야 한다.

# 설계 문서 표준

- 모든 설계 문서는 Markdown 형식 사용
- 각 메카닉은 `design/gdd/`에 전용 문서 보유
- 문서에는 아래 8개 필수 섹션 포함:
  1. **개요** — 한 단락 요약
  2. **플레이어 판타지** — 의도하는 감각과 경험
  3. **상세 규칙** — 명확한 메카닉 기술
  4. **공식** — 변수와 함께 수학 정의
  5. **엣지 케이스** — 특수 상황 처리 방법
  6. **의존성** — 연관 시스템 목록
  7. **조정 항목** — 설정 가능한 수치 식별
  8. **수용 기준** — 테스트 가능한 성공 조건
- 밸런스 수치는 출처 공식 또는 근거에 링크 필수

# 테스트 표준

## 스토리 유형별 테스트 증거

모든 스토리는 완료 처리 전 적합한 테스트 증거 필요:

| 스토리 유형 | 필요 증거 | 위치 | 게이트 수준 |
|---|---|---|---|
| **로직** (공식, AI, 상태 머신) | 자동화 단위 테스트 — 통과 필수 | `tests/unit/[system]/` | 차단 |
| **통합** (다중 시스템) | 통합 테스트 또는 플레이테스트 문서 | `tests/integration/[system]/` | 차단 |
| **비주얼/느낌** (애니메이션, VFX) | 스크린샷 + 리드 승인 | `production/qa/evidence/` | 권고 |
| **UI** (메뉴, HUD, 화면) | 수동 워크스루 문서 또는 인터랙션 테스트 | `production/qa/evidence/` | 권고 |
| **설정/데이터** (밸런스 튜닝) | 스모크 체크 통과 | `production/qa/smoke-[date].md` | 권고 |

## 자동화 테스트 규칙

- **네이밍**: 파일은 `[system]_[feature]_test.[ext]`, 함수는 `test_[scenario]_[expected]`
- **결정론**: 테스트는 매 실행마다 동일한 결과 — 랜덤 시드, 시간 의존 검증 금지
- **격리**: 각 테스트는 자체 상태를 설정·정리; 실행 순서 의존 금지
- **하드코딩 데이터 금지**: 테스트 픽스처는 상수 파일 또는 팩토리 함수 사용
  (예외: 정확한 수치 자체가 요점인 경계값 테스트)
- **독립성**: 단위 테스트에서 외부 API, DB, 파일 I/O 호출 금지 — 의존성 주입 사용

## 자동화 제외 항목

- 시각적 충실도 (셰이더 출력, VFX 외관, 애니메이션 곡선)
- "느낌" 품질 (입력 반응성, 체감 무게감, 타이밍)
- 플랫폼별 렌더링 (헤드리스가 아닌 대상 하드웨어에서 테스트)
- 전체 게임플레이 세션 (자동화가 아닌 플레이테스트로 커버)

## CI/CD 규칙

- 자동화 테스트 스위트는 main 푸시 및 모든 PR마다 실행
- 테스트 실패 시 머지 불가 — 테스트는 CI의 차단 게이트
- CI 통과를 위해 실패한 테스트 비활성화·스킵 금지 — 근본 원인 수정
- 엔진별 CI 명령어:
  - **Godot**: `godot --headless --script tests/gdunit4_runner.gd`
  - **Unity**: `game-ci/unity-test-runner@v4` (GitHub Actions)
  - **Unreal**: headless runner with `-nullrhi` flag
